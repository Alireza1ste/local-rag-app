# local-rag-app Instructions

These instructions guide development in the local-rag-app RAG (Retrieval-Augmented Generation) project using LangChain, Ollama, and Gradio.

## Code Style & Formatting

### Type Hints (Mandatory)
- All function signatures **must** include complete type hints for parameters and return types
- Use `Optional[T]` for nullable types (prefer over `T | None` for Python 3.9 compatibility)
- Use `Sequence[T]` for collections that may be lists, tuples, or other sequences
- Example:
  ```python
  def build_rag_chain(
      llm: ChatOllama, 
      retriever: BaseRetriever, 
      role: str
  ) -> Runnable[dict, str]:
      """Build and return a RAG chain."""
  ```

### Function Length (Strict Limit: 30 Lines)
- **Every function must be ≤ 30 lines** (excluding docstrings and comments)
- If a function exceeds 30 lines, extract sub-functions with clear responsibilities
- **Rare Exceptions Only**:
  - Gradio interface definitions (complex UI layouts)
  - Complex LangChain chain configurations with multiple `.pipe()` operations
- Use helper functions with descriptive names rather than inline logic
- Measure: Count actual code lines (not blanks or comments)

### Use Pathlib Over os.path
- Always import and use `pathlib.Path` for file operations
- ❌ Avoid: `os.path.join()`, `os.path.exists()`, string path manipulation
- ✅ Use: `Path("/some/path") / "file.txt"`, `path.exists()`, `path.read_text()`

### Use Dataclasses for Configuration
- Use `@dataclass` for configuration objects and data transfer objects
- Define at module top, before functions
- Example:
  ```python
  from dataclasses import dataclass
  
  @dataclass
  class RAGConfig:
      model_name: str
      temperature: float = 0.3
      top_k: int = 5
  ```

## Architecture & File Organization

### Current State (Monolithic)
- app.py is currently 343 lines with all logic (UI, RAG, document processing, helpers)

### Refactoring Target
- **When implementing new features or major refactors**, organize into logical modules:
  - `models.py` - Data structures, config dataclasses, Pydantic models
  - `rag_chain.py` - RAG pipeline construction and core logic
  - `document_processor.py` - Document loading, parsing, text splitting
  - `app.py` - Gradio UI orchestration and main entry point
- Keep imports organized: stdlib → third-party → local modules
- No circular imports; use dependency injection where needed

### Module-Level Constants
- Pre-compile regex patterns at module level (as already done with `QUESTION_PREFIX_RE`)
- Define all constants in UPPERCASE before any function definitions
- Document constants with brief comments explaining their purpose

## LangChain/ML-Specific Patterns

### Builder Functions
- Use factory functions (`build_*`) to construct LangChain components
- Keep builders focused: one builder per component type (llm, embeddings, rag_chain, retriever)
- Example: `build_llm()`, `build_embeddings()`, `build_rag_chain()`

### Error Handling in LangChain Chains
- Wrap LLM calls and retrieval in try-except blocks
- Return tuples of (success_flag, result_or_error_message) from RAG operations
- Use specific exception types where possible (avoid bare `except Exception`)

### Document Processing
- Use LangChain loaders: `TextLoader`, `PyPDFLoader`
- Always apply `RecursiveCharacterTextSplitter` for consistent chunking
- Store Documents with metadata (source, chunk index) for traceability

## Documentation & Comments

### Docstrings (Required for All Public Functions)
- Use Google-style docstrings for clarity and IDE support
- Include Args, Returns, and Raises sections
- Example:
  ```python
  def condense_question(history: list[dict], message: str) -> str:
      """Condense a question using chat history context.
      
      Args:
          history: List of previous conversation turns.
          message: Current user message to process.
      
      Returns:
          Condensed question string, or empty string if invalid input.
      
      Raises:
          ValueError: If history is malformed.
      """
  ```

### In-Line Comments
- Use `# OPTIMIZATION:` prefix for performance notes
- Use `# Fixed:` prefix for bug fix comments
- Use `# TODO:` for future improvements
- Avoid obvious comments; only explain *why*, not *what*

### README Structure
- Include: Project purpose, setup instructions, configuration examples, usage examples
- Document environment variables (create `.env.example`)
- Add troubleshooting section

## Testing Practices

### Test Location
- Create `tests/` directory at project root
- Test files follow pattern: `test_<module>.py`
- Parallel structure to `src/local_rag_chat/`

### Pytest Fixtures
- Use conftest.py for shared fixtures (mock LLM, sample documents, etc.)
- Fixtures for: `sample_documents`, `mock_llm`, `mock_qdrant_store`

### What to Test
- Document processing (loading, splitting, embedding)
- RAG chain construction and chain invocation
- Question condensation logic
- Error handling paths

### Example:
```python
def test_build_rag_chain_returns_runnable(mock_llm, mock_retriever):
    """Verify RAG chain is properly constructed."""
    chain = build_rag_chain(mock_llm, mock_retriever, "Test role")
    assert hasattr(chain, "invoke")
```

## Error Handling

### Consistent Pattern
- Use German user-facing messages (maintains current UX)
- Use English internal logging/exception messages
- Always return status tuple: `(success: bool, message_or_result: str)`
- Catch specific exceptions where possible; use descriptive error messages

### Error Message Format
```python
try:
    result = llm.invoke(input_data)
except ValueError as e:
    return False, f"Invalid input: {str(e)}"
except Exception as e:
    logger.error(f"Unexpected error: {e}", exc_info=True)
    return False, "❌ An unexpected error occurred. Check logs."
```

### User Feedback
- Use emoji indicators: ✅ success, ❌ error, ⚠️ warning
- Keep error messages concise and actionable
- Log full stack traces internally; show user-friendly messages externally
